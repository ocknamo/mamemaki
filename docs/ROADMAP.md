# ロードマップ

Split LN Sender は MVP として「複数の Lightning Address へ NWC で順番に送金する」
最小機能を実装済みです([DESIGN.md](./DESIGN.md) 参照)。本ドキュメントは
**MVP で意図的に対応しなかった機能**と、レビューで**既知の制限**とした項目を、
今後の実装候補としてフェーズ別に整理したものです。

優先度は「資金の安全性 > 対象ユーザーの中心的ユースケース > 利便性」の順で付けています。
各項目は独立して着手できます。

---

## MVP で実装済み(参考)

- 送金先リスト(行の追加・編集・削除)、CSV貼り付け入力、合計表示
- NWC 接続(NIP-47、info イベント検証つき)、逐次送金、進捗・結果表示
- 信頼境界の検証: リレー応答の署名/pubkey/`#e` 検証、invoice 金額検証(LUD-06)、
  `unconfirmed`(成否不明)状態の区別、送金中の編集ロック、バッチのキャンセル

---

## Phase 1 — 資金の安全性を高める

MVP でも致命的な誤送金は防いでいるが、さらに「二重払い/誤額」を減らすための強化。

| 項目 | 内容 | 現状 |
|------|------|------|
| `lookup_invoice` による自動再確認 | `unconfirmed` の行を NWC `lookup_invoice` で自動照会し、成立/未成立を確定させる。二重払いの主要因を根本から解消 | 未対応(ユーザーに手動確認を促すのみ) |
| `payment_hash` 照合 | bolt11 を完全デコードして `payment_hash` を取り出し、`sha256(preimage) === payment_hash` で支払い成立を暗号的に検証 | 未対応(preimage の存在確認まで) |
| `description_hash` 検証 | LUD-16 metadata の SHA-256 と invoice の `description_hash` を照合し、invoice のすり替えを検知 | 未対応 |
| 複数リレー対応 | NWC URI の複数 `relay` パラメータすべてに接続し、単一リレー障害での全機能停止を回避 | 未対応(最初の1つのみ使用) |

## Phase 2 — 分配ロジック(中心的ユースケース)

「投げ銭・一括送金」を楽にするための、金額入力を補助する機能群。要件の
「対応しないもの」に挙がっていたが、対象ユーザー(イベント主催者・コミュニティ送金)の
中核ニーズに近い。

| 項目 | 内容 |
|------|------|
| 均等割り | 合計額を入力し、送金先で等分する |
| 比率(weight)送金 | 送金先ごとに重みを指定し、合計額を按分する |
| Split Zap 読込 | Nostr の Zap split 設定(NIP-57 の `zap` タグ等)を読み込んで送金先・比率を自動生成 |

## Phase 3 — Nostr 連携(入力手段の拡張)

Lightning Address の直接入力以外に、Nostr エコシステムから送金先を取り込む。

| 項目 | 内容 |
|------|------|
| Nostr npub 入力 | `npub`/`nprofile` からプロフィールの `lud16`(Lightning Address)を解決 |
| Nostr イベント読込 | 特定イベントのリアクション/リプライ参加者などを送金先として取り込む |

## Phase 4 — 暗号化・接続の将来対応

| 項目 | 内容 |
|------|------|
| NIP-44 暗号化 | NIP-04 は非推奨化が進行中。info イベントの `encryption` を見て NIP-44 を優先し、NIP-04 のみのウォレットへはフォールバック |
| WebLN 対応 | NWC に加え、ブラウザ拡張の WebLN を送金経路として選択可能に |
| WalletConnect 以外の接続 | LNC(Lightning Node Connect)等、他の接続方式 |

## Phase 5 — 利便性・データ管理

MVP では「LocalStorage のみ・保存最小限」に絞った領域。

| 項目 | 内容 |
|------|------|
| 送金先リストの保存 | 入力した送金先リストを LocalStorage に保存・復元 |
| 送金履歴 | 過去のバッチ送金の結果を保存し、後から参照 |
| CSV エクスポート | 送金先リストや結果を CSV で書き出し |
| QRコード読取 | NWC URI や Lightning Address を QR から読み取る |
| URL 共有 | 送金先リストを URL に載せて共有(取り扱いに注意が必要なため要検討) |

## 対応予定なし(または慎重に検討)

- **並列送金**: 逐次送金は進捗の分かりやすさ・レート制限回避・二重払いリスク低減の
  観点から意図的な設計。並列化は当面予定しない。
- **URL 共有での secret 混入**: NWC secret を URL に載せることは絶対に避ける。
  送金先リストの URL 共有を実装する場合も secret は対象外とする。

---

## 補足: 現状の技術的前提

- サーバーレス / バックエンドなし、LocalStorage のみ、追加依存は最小限
  (`@noble/curves` のみ)という MVP の方針は今後も維持する。
- 上記のうち Phase 1 の `payment_hash` / `description_hash` 照合と Phase 4 の
  NIP-44 は bolt11 の完全デコードや追加の暗号処理を要するため、依存追加の
  是非を含めて設計時に判断する。
